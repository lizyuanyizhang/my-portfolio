# 「我的关注」聚合页方案评估

> 页面名称：我的关注 | Follow Me | Folgen  
> 目的：集中展示社交媒体、Newsletter、RSS、播客等入口，降低首页信息密度。

---

## 一、可填入的内容格式

### 1. 支持的入口类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **社交媒体** | 可关注/连接的平台 | X (Twitter)、小红书、即刻、LinkedIn、GitHub、Instagram、Mastodon、Bluesky |
| **播客** | 播客订阅链接 | 小宇宙、Apple Podcasts、Spotify |
| **Newsletter** | 邮件订阅 | Substack、Buttondown、Revue、自定义订阅页 |
| **RSS** | 博客/文章订阅 | 本站 RSS、外部博客 RSS |
| **视频** | 视频频道 | YouTube、B 站 |
| **其他** | 任意可分享的链接 | Discord、Telegram、个人站点等 |

### 2. 单条数据建议结构

每条「关注入口」建议包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| **名称** | ✓ | 如「小宇宙播客」「Newsletter」 |
| **链接** | ✓ | 目标 URL |
| **类型** | 建议 | 用于分组：social / podcast / newsletter / rss / video / other |
| **图标** | 可选 | emoji（🎙️）或 lucide 图标名 |
| **描述** | 可选 | 简短说明，如「关于生活与工作的思考」 |

---

## 二、数据来源方案对比

### 方案 A：Notion 数据库同步（推荐）

**流程**：Notion 数据库 → Elog sync → `content/elog-follow/` → 转换脚本 → `data.zh.json` 的 `followLinks` 数组

| 优点 | 缺点 |
|------|------|
| 可视化维护，手机也能改 | 需新增 elog 配置与脚本 |
| 与现有随笔/摄影/影像流程一致 | GitHub Actions 需增加一项 Secrets |
| 可随时增删、调整顺序 | 首次搭建略多一步 |
| 支持分组、描述等结构化信息 | |

**Notion 数据库结构建议**：

| 列名 | Notion 类型 | 对应字段 |
|------|-------------|----------|
| 名称 | Title | platform / name |
| 链接 | URL | url |
| 类型 | Select | type（社交媒体/播客/Newsletter/RSS/其他） |
| 图标 | Text | icon（emoji，如 🎙️） |
| 描述 | Text | description（可选） |

**多语言**：名称、描述可走翻译（同随笔/影像）；或仅在中文维护，英文/德文用映射或简版文案。

---

### 方案 B：复用 personalInfo + 扩展

**思路**：首页的「联系我」已包含 linkedin、xiaohongshu、github、x、podcast、jike 等。在 `personalInfo` 中新增 `followLinks` 数组，或直接复用现有字段。

| 优点 | 缺点 |
|------|------|
| 无需 Notion，零额外同步 | personalInfo 会变臃肿 |
| 实现快，改动小 | Newsletter、RSS 等需新字段，结构不统一 |
| | 首页联系与关注页语义不同（联系 vs 订阅） |

**适用**：若「我的关注」与首页联系高度重合，且不打算频繁增减，可考虑。

---

### 方案 C：纯静态 JSON

**思路**：在 `data.zh.json` 中新增 `followLinks` 数组，手动编辑。

| 优点 | 缺点 |
|------|------|
| 实现最简单 | 每次改都要动代码/JSON |
| 无外部依赖 | 无可视化界面 |
| | 多语言需手动维护 en/de |

**适用**：链接很少、几乎不更新时可用。

---

## 三、推荐方案：Notion 同步（方案 A）

### 理由

1. **与现有架构一致**：沿用随笔、摄影、影像的 Notion + Elog 工作流，心智负担小。
2. **扩展性好**：以后加 Newsletter、RSS、新平台只需在 Notion 加一行。
3. **维护方便**：手机、网页都能改，无需碰代码。
4. **可分组展示**：按类型（社交媒体 / 播客 / Newsletter / RSS）分区展示。

### 实现步骤概要

1. 在 Notion 创建「我的关注」数据库，列结构如上。
2. 新建 `elog-follow.config.cjs`、`.elog-follow.env`，接入 Elog。
3. 新建 `scripts/notion-to-follow.cjs`，将 md/frontMatter 转为 `followLinks`。
4. 新建 `/follow` 页面，读取 `followLinks` 并按类型分组展示。
5. 在导航栏「留言」右侧新增「我的关注」入口。

### 与 personalInfo 的关系

- **首页「联系我」**：保留，继续用 personalInfo，侧重联系方式（邮箱、微信等）。
- **「我的关注」页**：用 `followLinks`，侧重关注/订阅（社交、播客、Newsletter、RSS）。
- 两者可部分重叠（如都展示播客链接），但语义和入口不同：联系 vs 关注。

## 四、多语言策略

| 策略 | 说明 |
|------|------|
| **全量翻译** | 名称、描述都走翻译 API，与随笔/影像一致 |
| **仅名称映射** | 常见平台名做固定映射（小宇宙→Xiaoyuzhou Podcast→Xiaoyuzhou Podcast） |
| **纯中文** | 暂不翻译，en/de 直接展示中文（实现最简单） |

建议：先用「仅名称映射 + 描述可选翻译」，复杂平台名再考虑接入翻译。

---

## 五、小结

| 维度 | 建议 |
|------|------|
| **数据来源** | Notion 数据库同步（方案 A） |
| **单条结构** | 名称、链接、类型、图标(可选)、描述(可选) |
| **支持格式** | 社交、播客、Newsletter、RSS、视频、其他链接 |
| **多语言** | 先做平台名映射，再视需要加描述翻译 |

若确认采用 Notion 方案，可在此基础上细化 Notion 列、Elog 配置和前端展示结构。
