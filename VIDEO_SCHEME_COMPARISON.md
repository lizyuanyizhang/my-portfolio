# 影像页方案对比 / Video Page Scheme Comparison

> 目标：在影像菜单下展示你创作的视频作品（记录性质、AI 生成等），需永久性存储，且设计风格与应用菜单一致。
>
> ✅ **方案 A（YouTube/B 站嵌入）已实现** — 见 `data.json` 的 `videos` 数组、`src/pages/Video.tsx`

---

## 一、UI 设计（统一方案）

无论选择哪种存储，**页面布局与视觉**将与应用页保持一致：

| 元素 | 设计 |
|------|------|
| 布局 | 紧凑网格：`grid-cols-2 ~ 6`，响应式 |
| 卡片 | `rounded-xl`，`aspect-video` 封面/预览，`border border-ink/5` |
| 动效 | `hover:scale-105`，`transition-all duration-300` |
| 内容 | 标题、描述、标签（记录 / AI 生成 / 旅行等）、日期 |
| 交互 | 点击卡片 → 弹窗或详情页播放，或跳转外链 |

**数据结构**（与 `projects` 类似，新增 `videos`）：

```json
{
  "videos": [
    {
      "id": "1",
      "title": "上海街头的一天",
      "description": "用手机记录的 citywalk 碎片。",
      "cover": "/images/videos/cover1.jpg",
      "videoUrl": "https://...",
      "duration": "3:42",
      "tags": ["记录", "城市"],
      "date": "2024-12",
      "source": "self"
    }
  ]
}
```

## 二、存储方案对比

### 方案 A：YouTube / Bilibili 嵌入（推荐起步）

| 项目 | 说明 |
|------|------|
| **存储方式** | 视频上传至 YouTube 或 B 站，在页面中嵌入播放 |
| **永久性** | 依赖平台规则，通常可长期使用 |
| **成本** | 完全免费 |
| **实施** | `videoUrl` 存 embed 链接，用 `<iframe>` 或 lite-youtube 等 |
| **优点** | 零成本、不占站内流量、播放器成熟、自动转码 |
| **缺点** | 依赖第三方平台、可能有品牌/广告、链接变更需手动更新 |
| **适合** | 记录类、AI 生成短片、数量不多、优先控制成本 |

---

### 方案 B：Supabase Storage（与现有留言逻辑一致）

| 项目 | 说明 |
|------|------|
| **存储方式** | 视频文件上传至 Supabase Storage `videos` 桶 |
| **永久性** | 由 Supabase 项目生命周期决定 |
| **成本** | 免费约 1GB；视频 10MB～100MB 不等，约可存 10～100 个 |
| **实施** | 新建 `videos` 桶，上传后拿公开 URL 写入 `data.json` 或数据库 |
| **优点** | 已接入、统一后端、文件完全自管、无平台品牌 |
| **缺点** | 免费额度有限、大视频易占满、需要压缩/转码 |
| **适合** | 短视频为主、视频数量有限、希望全部自托管 |

---

### 方案 C：静态资源 + GitHub 仓库

| 项目 | 说明 |
|------|------|
| **存储方式** | 视频放在 `public/images/videos/` 或 `public/videos/` |
| **永久性** | 随仓库和 Pages 部署存在 |
| **成本** | 免费（受仓库容量限制） |
| **实施** | 本地放视频，路径如 `/videos/xxx.mp4` |
| **优点** | 实现简单、无第三方服务、加载路径清晰 |
| **缺点** | GitHub 单文件 ≤100MB、仓库建议 <1GB、拉大构建体积 |
| **适合** | 少量、短、已压缩的演示视频 |

---

### 方案 D：Cloudinary 视频托管

| 项目 | 说明 |
|------|------|
| **存储方式** | 上传至 Cloudinary，使用其视频 URL |
| **永久性** | 由 Cloudinary 账户与用量决定 |
| **成本** | 免费约 25 积分/月，可支撑少量视频 |
| **实施** | 使用 Cloudinary 的 URL API，`videoUrl` 指向其地址 |
| **优点** | 面向视频优化、自动转码、CDN、可做缩略图等变换 |
| **缺点** | 需额外服务、免费额度有限 |
| **适合** | 重视画质、需要多种清晰度、愿意管理新平台 |

## 三、综合对比表

| 维度 | A 平台嵌入 | B Supabase | C 静态文件 | D Cloudinary |
|------|-----------|------------|------------|--------------|
| 成本 | 免费 | 免费（额度有限） | 免费 | 免费（额度有限） |
| 永久性 | 依赖平台 | 较稳定 | 依赖仓库 | 较稳定 |
| 实施难度 | 低 | 中 | 低 | 中 |
| 视频大小 | 基本不限 | 受 1GB 限制 | 单文件 ≤100MB | 受积分限制 |
| 自管程度 | 低 | 高 | 最高 | 中 |
| 现有依赖 | 无 | 已有 Supabase | 无 | 需新配置 |

## 四、建议选择

1. **优先试水**：选 **方案 A（YouTube/B 站）**  
   - 成本低，上线快  
   - 适合「记录 + AI 生成」的初期展示  

2. **全部自管**：选 **方案 B（Supabase）**  
   - 与留言等功能统一  
   - 需控制视频数量与大小  

3. **极简自托管**：选 **方案 C**  
   - 适合 3～5 个短视频、已压缩好  

4. **长期专业需求**：选 **方案 D**  
   - 画质与体验要求高时再考虑
